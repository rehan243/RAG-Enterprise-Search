## RAG Retrieval Tradeoffs
When building a RAG system, I've found that a combination of techniques is necessary for effective retrieval. **Chunking** is a crucial step, as it determines the granularity of the information retrieved. I've seen cases where chunking at the sentence level results in too much redundancy, while chunking at the paragraph level loses context. To mitigate this, I use a hybrid dense + sparse search approach, leveraging both semantic search (e.g., using a transformer-based embedding model) and keyword search (e.g., using BM25). However, this can lead to a large number of results, which need to be **re-ranked** to prioritize the most relevant ones. I've observed that re-ranking the top 100 results using a cross-encoder model can improve precision by 20-30% compared to relying solely on the initial search scores. A key gotcha is that the re-ranking model needs to be fine-tuned on the specific dataset to achieve optimal results.
